# MCPSnitch value proof

Status: tested value statement  
Date tested: 2026-06-14  
Version: v0.1.6

## The value in one sentence

MCPSnitch gives an MCP user a one-command, silent-by-default runtime witness for local stdio MCP servers: it keeps clean sessions quiet, produces tamper-evident evidence, and speaks only when the OS process observer sees behavior outside the server's declared profile or when that stronger observer is unavailable.

## Who gets value

- **Individual engineers using Claude Code / MCP servers:** keep MCP servers installed without losing visibility into unexpected local file or network behavior.
- **Security-minded teams evaluating MCP adoption:** get a reproducible audit trail and false-positive measurements before allowing more third-party MCP servers.
- **MCP client/plugin authors:** get a concrete, open evidence layer they can inspect, wrap, or build into a larger control plane.

## What problem it solves

MCP servers commonly run as local commands such as `npx ...`. They can read files, open network sockets, and handle secrets while the agent user mostly sees only the final model response. Static config review alone does not answer the runtime question: **what did this server process actually do during this session?**

MCPSnitch answers that question for local stdio MCP servers with two explicitly separated layers:

1. **JSON-RPC heuristic tripwire**: records visible MCP `tools/call` traffic and cheap structural findings. This is useful but evadable.
2. **Process observer evidence**: samples the wrapped process tree with `lsof` and profiles OS-observed files/sockets against expected server behavior. This is stronger evidence, still sampled rather than syscall-complete.

## Actual live-use verdict

On 2026-06-14, the released command was run the way a user would actually get it:

```bash
node scripts/live-value-check.mjs --release github:rudycelekli/mcpsnitch#v0.1.6
```

That script invokes `npx -y github:rudycelekli/mcpsnitch#v0.1.6 run -- ...` and drives real stdio MCP traffic through the wrapper.

Results:

- **Clean real MCP session:** wrapped `@modelcontextprotocol/server-filesystem@2026.1.14`, initialized it, listed tools, called `list_directory`, and called `read_text_file`.
  - MCPSnitch alert lines: **0**
  - Tool calls recorded: **2**
  - OS process observations recorded: **18**
  - Medium/high findings: **0**
  - Witness chain: **verified** (`27` entries)
  - Note: the filesystem server wrote its own startup lines to stderr; MCPSnitch itself did not alert.
- **Intentional live violation:** wrapped a real stdio MCP process that opened and held an unexpected local TCP socket under the `filesystem` profile.
  - MCPSnitch alert lines: **1**
  - Alert rule: `observed_unexpected_network_connection`
  - Alert source: `process_observer`
  - Tool calls still proxied successfully: **1**
  - Witness chain: **verified** (`15` entries)
- **No process observer honesty path:** ran with `--no-process-observer`.
  - MCPSnitch alert lines: **1**
  - Alert rule: `process_observer_unavailable`
  - Witness chain: **verified** (`3` entries)

Live value verdict: **yes, MCPSnitch is valuable for local stdio MCP users today** because the user-visible behavior is right: normal filesystem MCP work continues without MCPSnitch noise, but an OS-observed profile violation becomes one actionable alert with a verifiable audit trail. The value is not theoretical; it appeared in an actual wrapper run.

The honest boundary remains: the value is strongest for local stdio MCP servers with sustained enough file/socket behavior for sampled `lsof` to observe. Remote HTTP/SSE servers and sub-interval operations are still explicitly outside v0.1.x's strongest evidence.

Observed behavior:

- `filesystem` was assigned `filesystem` and wrapped.
- `fetcher` was assigned `fetch` and wrapped.
- `unknown_local` was assigned conservative `generic` and wrapped.
- `remote_sse` was marked `remote_http_or_sse_not_process_observable` and was **not** wrapped.

- `uninit` restored wrapped stdio entries and created its own backup before unwrapping.

## What makes it different

MCPSnitch is not just a proxy log and not a static scanner. Its differentiated value is the correlation of:

- what the MCP client visibly asked the server to do;
- what the server process tree actually did at the OS layer;
- what profile expectation that behavior matched or violated;
- whether the audit trail still verifies after the session.

That combination is what turns it into a runtime evidence layer rather than another MCP checklist.

## What it deliberately does not claim

- It does **not** prevent exfiltration.
- It does **not** sandbox or block syscalls.
- It does **not** claim complete coverage of remote HTTP/SSE MCP servers.
- It does **not** catch every short-lived socket/file open; `lsof` is sampled and can miss sub-interval behavior.
- It does **not** treat JSON-RPC heuristics as proof of OS behavior.

These limitations are part of the value: the tool is useful because it is explicit about what it can prove and what it cannot.

## How to demonstrate value to a user

```bash
/plugin marketplace add rudycelekli/mcpsnitch
/plugin install mcpsnitch
/mcpsnitch:mcpsnitch-init
```

Then run normal MCP-backed agent work. On clean sessions MCPSnitch should say nothing. When the user wants proof:

```bash
npx -y github:rudycelekli/mcpsnitch#v0.1.6 report --json
npx -y github:rudycelekli/mcpsnitch#v0.1.6 verify --json
```

The product moment is not a dashboard. It is the first time an MCP server opens an unexpected socket or sensitive file and MCPSnitch emits one actionable alert with verifiable evidence.
